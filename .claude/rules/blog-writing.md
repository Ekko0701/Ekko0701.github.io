## 1. 말투 (Tone)

### 기본 원칙
**서술체 (`~다` 체)**를 사용한다.

```
# 올바른 예
DHCP는 자동으로 IP 주소를 할당해주는 프로토콜이다.
BFS를 사용하면 레벨 순서로 탐색할 수 있다.

# 잘못된 예 (사용 금지)
DHCP는 자동으로 IP 주소를 할당해주는 프로토콜임
정리해봄, 공부한 내용임
```

### 도입부
- 1~3문장으로 주제를 소개한다.
- "오늘은 ~을 정리해봄" 같은 일기체 표현은 사용하지 않는다.
- 독자가 이 글에서 얻을 수 있는 것을 간결하게 제시한다.

```
# 올바른 예
Docker는 애플리케이션을 컨테이너 단위로 패키징하여 어디서든 동일하게 실행할 수 있게 해주는 플랫폼이다.
이 글에서는 Docker의 핵심 개념과 자주 사용하는 명령어를 정리한다.
```

### 허용 표현
- 독자에게 직접 말을 건네는 경우: "~해보자", "~살펴보자" 허용

## 2. Front Matter

모든 포스트에 아래 필드를 반드시 포함한다.

```yaml
---
title: "제목"
categories:
  - 카테고리명
tags:
  - 태그1
  - 태그2
---
```

### 제목 규칙

| 카테고리 | 형식 | 예시 |
|---|---|---|
| Algorithm | `[플랫폼] 번호. 문제이름` | `[Leetcode] 102. Binary Tree Level Order Traversal` |
| CS / iOS / Java / Spring / Others | `핵심개념 - 부연설명` | `DHCP와 NAT - 자동 IP 할당과 주소 변환` |

### Tags 규칙
- 핵심 기술 키워드만 포함 (최소 1개, 최대 5개)
- 공식 표기를 따름: `BFS`, `Docker`, `SwiftUI`, `Spring MVC`

## 8. 참고 자료 우선순위

블로그 글 작성 시 아래 순서로 참고 자료를 우선적으로 활용한다.

1. **공식 문서 (최우선)** — Apple Developer Documentation, Spring 공식 문서, MDN, 언어/프레임워크 공식 레퍼런스
2. **Swift Evolution / 공식 제안서** — Swift 언어 변경 사항은 SE 문서 기준으로 서술
3. **공식 GitHub 저장소** — 라이브러리 소스 코드, 예제, 릴리스 노트
4. **기술 블로그 / 서드파티 자료** — 공식 문서로 확인 불가한 내용에 한해 보조적으로 참고

### 작성 전 공식 문서 확인 (필수 절차)

글을 **작성하기 전에** 반드시 관련 공식 문서를 먼저 조회한다. 학습된 지식이나 기억에만 의존해 작성하지 않는다.

1. 본문을 쓰기 전, 주제와 관련된 공식 문서(위 우선순위 1~3순위)를 `WebFetch` / `WebSearch`로 직접 조회한다.
2. API 시그니처, enum 케이스, 메서드 이름, 동작 방식, 최신 추가 사항(신규 API·버전별 변경)을 공식 문서와 대조하여 사실을 확인한다.
3. 확인되지 않은 내용은 추측해서 서술하지 않는다. 불확실하면 명시하거나 제외한다.
4. 작성 후에도 코드 예시의 API와 핵심 주장을 공식 문서 기준으로 재검증한다.

> 기억에 의존해 먼저 쓰고 나중에 검증하는 순서를 금지한다. **공식 문서 조회 → 작성 → 재검증** 순서를 지킨다.

### 포스트 하단 참고 링크 규칙

- 참고한 공식 문서 링크를 포스트 마지막에 반드시 포함한다.
- 형식:
  ```
  **참고**
  - [Apple Developer Documentation - 주제명](URL)
  - [SE-XXXX - 제안서 제목](URL)
  ```
- 공식 문서 없이 서드파티 자료만 참고한 경우 해당 사실을 포스트 내에 명시한다.

## 9. 금지 사항

- `~임`, `~함`, `~봄` 등 반말체 종결 금지
- 일기체 도입부 금지 ("오늘은 ~을 정리해봄")
- `tags` 누락 금지
- `categories` 누락 금지
- 공식 문서 확인 없이 기억에만 의존한 작성 금지 (8번 "작성 전 공식 문서 확인" 절차 준수)
